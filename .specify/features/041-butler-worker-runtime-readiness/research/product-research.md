# Product Research: Feature 041 Butler / Worker Runtime Readiness + Ambient Context

## 用户视角问题

本次 live usage 暴露的问题不是“模型答错一句话”这么简单，而是产品承诺和默认行为之间出现了断层：

1. 用户已经被告知 OctoAgent 有 Butler、能创建 Worker、能使用工具、能联网查资料。
2. 但在最普通的日常问题上，例如“今天天气怎么样”“今天几号”“帮我查一下官网”，系统仍表现得像一个没有当前时间、没有网络、不会委派的 stateless chat shell。
3. 对普通用户来说，这会直接推翻他们对“Personal AI OS”的基本信任，因为这类问题不是高级场景，而是进入长期助手心智模型的最低门槛。

## 为什么这不是小修小补

如果只补一句 prompt 或只补一个天气工具，会留下三个产品层面的隐患：

- 主 Agent 仍不知道什么时候应该派 Worker
- Worker 仍不知道自己可用的工具面和当前环境事实
- Project / Workspace / Tool 治理链仍没有进入“真实外部问题”的主路径

因此，这个问题必须作为一个 readiness feature 处理，而不是零散修正。

## 产品原则

1. 对“今天/现在”类问题，系统应默认知道当前时间，不应逼用户反复补充本机时间。
2. 对“天气/最新/官网/网页资料”类问题，系统应先判断缺不缺关键信息，再利用受治理执行面完成查询。
3. 用户应该能在 runtime truth 中看到：是谁去查的、用了什么工具、为什么有权限、为何退化。
4. 如果缺城市、缺网络或后端不可用，系统应说清楚“缺的是什么”，而不是笼统说自己没有能力。
