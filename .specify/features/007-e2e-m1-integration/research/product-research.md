# Feature 007 产品调研（product-only 视角补充）

## 目标

- 明确 Feature 007 对用户可见价值，不把它误做成“新功能开发”。
- 明确验收产物应偏“可靠性证据”，而不是“界面变化”。

## 用户价值

1. 对 Owner 的价值
- 降低“模块各自可用但组合失效”的系统性风险。
- 在进入 M1.5 前，把 M1 承诺的最小智能闭环先做实证闭环。

2. 对协作者的价值
- 提供可执行、可回归的集成测试基线，避免后续改动打破 004/005/006 契约。

## 范围建议

### 应该做

- 增加跨模块端到端测试（SkillRunner + ToolBroker + Policy/Approval）。
- 用验证报告把 Blueprint §14 的 M1 验收条目映射到证据。

### 不应该做

- 在 007 内直接引入新交互 UI 或大规模重构网关主链路。
- 为了“看起来完整”引入 M2 能力（如多渠道联动、交互式 shell 会话）。

## 参考实现对产品策略的启示

### OpenClaw

- `exec-approvals` 的产品形态强调“审批体验可解释+可追溯”，而非无条件自动化。
- 对 OctoAgent 的含义：007 要先把审批链路证据做好，再讨论更激进自动化策略。

### Agent Zero

- `skills_tool` 体现“能力模块化、按需加载”，产品上更像平台能力而非单次功能。
- 对 OctoAgent 的含义：007 应收敛成“平台能力联调里程碑”。

## 产品级风险

- 若 007 做成“半重构”而非“验收闭环”，会延迟 M1 结项并增加回归面。
- 若没有集成证据，后续 M1.5 推进会建立在不稳定基础上。

## 结论

Feature 007 的产品目标应定义为：

- “把 M1 的承诺变成可回放、可回归、可审计的证据闭环”，
- 而不是“继续堆叠新功能”。
