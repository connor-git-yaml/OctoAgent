# Feature 010 产品调研：Checkpoint & Resume Engine

**特性分支**: `010-checkpoint-resume-engine`
**调研日期**: 2026-03-03
**调研模式**: full（产品 + 技术 + 在线补充）

## 1. 目标重述

Feature 010 的业务目标不是“再做一个任务队列”，而是把 M1.5 的恢复语义闭环：

- 任务中断后从最后成功节点继续，而不是全量重跑
- 恢复过程可解释、可审计、可重试
- 恢复失败时系统仍保持安全终态，不出现静默失败或重复副作用

## 2. 用户价值

### Owner（主维护者）

- 降低长任务因重启/超时导致的重复劳动和重复风险动作
- 提升“可恢复”体验：失败后可以 resume，而不是只能重跑
- 排障路径更清晰：知道失败发生在哪个节点、哪次恢复尝试

### 协作者（未来扩展）

- 有稳定的恢复语义后，后续 Watchdog/Task Journal/Diagnostics 可以建立在一致的状态机上
- 形成可测试契约，减少后续 feature 串联时的语义漂移

## 3. 范围边界

### In Scope（本 Feature 必做）

- CheckpointSnapshot 领域模型与持久化结构
- Worker/Skill 节点边界的 checkpoint 写入策略
- Resume 引擎（加载、校验、恢复、继续执行）
- 恢复幂等保障（避免重复执行已落盘副作用）
- 事件可观测（checkpoint/resume 全链路事件）

### Out of Scope（本 Feature 不做）

- 不引入 Temporal/外部编排服务
- 不在本轮完成 Watchdog/Drift Detector（属于 Feature 011）
- 不做多 Worker 并发调度策略升级（依赖 Feature 008/009 后续演进）

## 4. 成功标准（产品视角）

1. 中断恢复路径可由用户复述：从“最近成功 checkpoint”继续。
2. 恢复失败有明确原因、可操作建议，不出现“任务消失”。
3. 恢复后不重复执行已确认副作用步骤。

## 5. 产品风险

- 风险 1：恢复逻辑过于“智能”导致状态不确定
  - 策略：先做确定性恢复（严格按 checkpoint + event 回放）
- 风险 2：恢复冲突语义不清（并发 resume、损坏快照）
  - 策略：显式冲突策略和事件类型，失败进入可审计终态
- 风险 3：范围膨胀（把 011/012 内容带入 010）
  - 策略：严格限定为“checkpoint + resume 引擎”

## 6. 结论

Feature 010 的定位合理且必要，属于 M1.5 闭环中的关键支点。建议按“最小可恢复核心”交付：先保证恢复正确性与幂等，再在 011/012 增强监控和诊断体验。
