# 产品调研报告: Feature 012 Logfire + Health/Plugin Diagnostics

**特性分支**: `codex/feat-012-logfire-health-plugin-diagnostics`
**调研日期**: 2026-03-03
**调研模式**: 在线+代码库

## 1. 需求概述

**需求描述**: 推进需求 012（M1.5 的 Logfire + Health/Plugin Diagnostics）。

**核心功能点**:
- 打通可观测链路（至少覆盖 Gateway 与 Tool/Model 关键路径）
- 增强健康检查，暴露关键子系统状态
- 增加插件/工具注册诊断，确保失败可见且主流程可降级

**目标用户**: OctoAgent 的 owner 与维护者（单用户、长期运行场景）。

## 2. 市场现状

### 市场趋势
- AI Agent 系统从“能跑”转向“可观测 + 可恢复 + 可治理”，健康诊断能力成为生产门槛。
- OTel 兼容链路（trace_id/span_id）已是主流方案，可降低后续平台迁移成本。

### 市场机会
- 当前 OctoAgent 已有事件系统与基础日志，补齐 trace 与诊断即可快速形成运维闭环。
- 与同类项目相比，OctoAgent 在“事件溯源 + 审批治理”有基础，012 可直接放大优势。

### 用户痛点
- 无法在单个入口快速判断“系统不可用是 DB/LLM/Runner/插件哪一层异常”。
- 工具注册失败时缺少结构化诊断，排障成本高。

## 3. 竞品分析

### 竞品对比表

| 维度 | OpenClaw | Agent Zero | AgentStudio | OctoAgent（目标） |
|------|---------|-----------|------------|------------------|
| 健康诊断入口 | `doctor` + gateway 健康检查 | 以运行日志为主 | 管理面板 + 执行器状态 | `/ready` + 子系统状态 + 诊断聚合 |
| 可观测链路 | OTel 扩展较完整 | 事件与日志偏工程化 | 平台化展示较强 | Logfire + structlog + Event 联动 |
| 插件/工具故障处理 | 强调 fail-open + diagnostics | 插件较少、偏内建 | 管理台策略为主 | 工具注册诊断 + 降级不中断 |
| 上手与排障体验 | 命令化强 | 开发者友好 | UI 友好 | API + 文档 + 测试证据 |

### 差异化机会

1. **事件与诊断统一**: 诊断结果可回写事件链，形成审计证据。  
2. **最小侵入升级**: 在不重构主链路前提下补齐观测与健康。  
3. **门禁前置化**: 在进入 M1.5 集成前先以 012 固化诊断能力，降低 013 风险。

## 4. 用户场景验证

### 核心用户角色

**Persona 1: 系统维护者（Connor）**
- 背景: 个人 AI OS Owner，负责功能迭代与故障恢复。
- 目标: 快速定位故障层级并判断是否可降级继续运行。
- 痛点: 当前健康信息与工具注册信息分散。

### 关键用户旅程

1. **上线前自检**: 调用 `/ready` 一次性确认关键子系统状态。  
2. **运行中排障**: 工具注册失败后查看 diagnostics，不阻断主链路。

### 需求假设验证

| 假设 | 验证结果 | 证据 |
|------|---------|------|
| 维护者需要可聚合健康状态 | ✅ 已验证 | `docs/blueprint.md` §12.3 + §14 M1.5 验收 |
| 插件/工具失败应 fail-open 并可诊断 | ✅ 已验证 | OpenClaw loader/doctor 设计 |
| Logfire/OTel 可作为统一观测基座 | ✅ 已验证 | Pydantic Logfire 文档 |

## 5. MVP 范围建议

### Must-have（MVP 核心）
- Logfire 初始化与环境开关可控（禁用不影响主流程）。
- `/ready` 增强为“核心检查 + 子系统检查 + 诊断摘要”。
- ToolBroker 提供 `try_register()` 与 `registry_diagnostics`。
- 针对上述能力补齐自动化测试。

### Nice-to-have（二期）
- 更细粒度 span 命名规范与跨进程 trace 透传。
- 诊断结果写入统一事件类型（如 `DIAGNOSTIC_*`）。

### Future（远期）
- 插件进程级隔离加载与自动修复建议（与 M1.5 should-have 对齐）。

### 优先级排序理由
可观测与诊断是 M1.5 进入 013 集成前的“先决稳定器”；先做这部分可以显著降低后续联调与验收不确定性。

## 6. 结论与建议

### 总结
需求 012 具备明确价值、低重构风险和高杠杆收益，适合在当前基线直接推进。

### 对技术调研的建议
- 明确 Logfire 在 FastAPI/HTTPX 下的最小可用埋点组合。
- 对齐 OpenClaw 的 fail-open + diagnostics 思路到 ToolBroker 注册流程。
- 明确 health 子系统检查项的默认集合和降级语义。

### 风险与不确定性
- 风险 1: 观测字段不统一（trace_id/span_id 混乱） -> 在测试中增加一致性断言。
- 风险 2: 健康检查过度耦合新组件 -> 通过可选状态检查与容错返回降低耦合。
