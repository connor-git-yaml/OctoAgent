# 产品调研：Agent Management Simplification

**特性分支**: `codex/050-agent-management-simplification`  
**日期**: 2026-03-14  
**范围**: `Agents` 页面信息架构、模板心智、主 Agent 管理、普通用户可理解的编辑体验  
**输入**: 当前 `AgentCenter / TemplateWorkspaceSection / ChatWorkbench`、`docs/blueprint.md`、本地同步的 Agent Zero `projects` 指南与 OpenClaw `channel-routing` 文档

## 1. 调研目标

围绕以下产品问题形成明确结论：

1. 普通用户应该如何理解“主 Agent”“其他 Agent”和“模板”
2. `Agents` 首页应该展示什么，哪些内容不应再占据主路径
3. 创建与编辑 Agent 时，哪些字段应以结构化控件表达，哪些应降级到高级设置
4. `Project` 在 Agent 管理中的地位应如何表达，才能保持心智稳定

## 2. 当前产品问题

### 2.1 模板、已创建对象和运行态混在一起

当前 `Agents` 页面把以下四类对象揉在同一个工作台里：

- 内置模板
- 已保存模板
- 模板编辑器
- 运行态与版本轨道

这会让普通用户很难回答最基本的问题：

- “我现在到底有几个 Agent？”
- “哪个是我正在用的主 Agent？”
- “我是在编辑模板，还是在编辑已经存在的 Agent？”
- “为什么这里会出现很多几乎一样的副本？”

### 2.2 页面默认语言仍偏开发者视角

当前页面存在较多只对开发者友好的表达，例如：

- `Worker 模板`
- `runtime kinds`
- `policy refs`
- `archetype`
- `tool profile`

这些词对系统设计者是准确的，但对普通用户不是有效的产品语言。

### 2.3 创建路径实际上在鼓励“复制副本”

当用户只想加 1-2 个 MCP 或调整一点工具边界时，当前主路径往往把动作表达成“复制一个新模板再发布”。  
这会把用户的真实意图“创建一个新 Agent”误导成“维护一套模板版本学”，最终带来一堆名字类似、描述类似、只差少量工具的对象。

### 2.4 项目归属没有被产品化表达

`Project` 在 OctoAgent 里已经是一等公民，但 `Agents` 页面目前没有把“每个 Agent 归属哪个项目、哪个是当前项目主 Agent”做成清晰的首屏事实。

## 3. 外部样本观察

### Agent Zero：Project 优先于杂乱配置

本地参考 `Agent Zero` 的 `projects` 指南可以看到，它把项目当作用户的第一层隔离单位：

- 先有 `Project`
- 再在项目内配置说明、记忆、凭证和 agent 行为
- 用户先理解“当前在哪个项目里”，再理解 agent 如何在该项目下工作

对本 Feature 最有价值的启发是：

- `Project` 是普通用户能理解的组织边界
- “创建/编辑”应该是一个明确的向导或表单，而不是一整页技术说明
- 项目内对象应尽量以列表和详情形式管理，而不是以模板/草稿/运行时混合工作台管理

### OpenClaw：运行时路由是系统真相，不应直接变成普通用户的主界面

OpenClaw 的文档清楚表达了两件事：

- 每次路由最终会选中一个默认 Agent 或绑定 Agent
- 复杂的路由和绑定规则存在，但并不意味着这些内部语义必须全部暴露为普通用户的日常管理界面

对本 Feature 的启发是：

- 普通用户看到的应该是“当前在用哪个 Agent”
- 绑定、路由、运行时 session 这些深层概念应由系统内部承载，必要时放到高级视图
- 主界面重点应回到“拥有的 Agent、用途、项目归属、可编辑项”

## 4. 产品结论

### 4.1 `Project` 应是 Agent 管理的第一层范围

普通用户最自然的心智不是“系统里有多少 profile”，而是：

- 我当前在哪个项目
- 这个项目有一个主 Agent
- 这个项目还可以有若干辅助 Agent

因此，`Agents` 页面默认应展示“当前项目的 Agent 列表”，而不是系统全量模板与 profile。

### 4.2 主 Agent 与其他内置 Agent 必须分开理解

产品上应明确区分：

- **主 Agent**：当前项目唯一、长期存在、可直接编辑保存、不可删除
- **内置模板**：创建新 Agent 时可选的起点，不是用户已拥有的 Agent
- **已创建 Agent**：用户基于模板或空白创建出来的真实对象，可编辑，可删除

### 4.3 首页应该是“列表 + 动作”，不是“说明 + 工作台”

`Agents` 首页最合理的信息架构应是：

1. 当前项目的主 Agent
2. 当前项目已创建的其他 Agent 列表
3. “新建 Agent” 主按钮
4. 每个 Agent 的 `编辑 / 删除` 动作

而不是默认进入“模板编辑器 + 说明卡片 + 运行态轨道”的复合工作台。

### 4.4 编辑页应以结构化控件为主

普通用户对以下内容不适合用自由文本输入：

- 使用的 LLM
- 默认工具组
- 固定工具
- MCP / Skill 能力绑定
- 所属项目

这些字段更适合：

- 单选
- 多选
- 搜索选择器
- checkbox / chip 选择

自由文本只保留给：

- 名称
- Persona / 用途说明
- 少量补充说明

### 4.5 高级信息必须降级

以下内容不应继续占据默认编辑页的中心区域：

- `runtime kinds`
- `policy refs`
- `tags`
- 原始 `tool profile` 术语
- “和 Agent Zero / OpenClaw 一样”这类设计者注释

它们应放入：

- 高级设置
- 诊断页
- 折叠区

## 5. 推荐信息架构

### `Agents` 首页

- 顶部：当前项目 + 主 Agent 摘要
- 主体：已创建 Agent 列表
- 行为：新建 Agent

每张 Agent 卡片只显示：

- 名称
- 用途摘要
- 所属项目
- 使用的模型别名
- 默认工具数 / 固定工具数
- 是否主 Agent
- `编辑 / 删除`

### “新建 Agent” 流程

1. 选择模板或空白起点
2. 填写名称与用途
3. 选择所属项目
4. 选择模型、默认工具组、固定工具
5. 如需能力绑定，再补 MCP / Skill 选择

### 编辑页

默认分成四个区块：

1. 基本信息
2. 模型与工具
3. 能力绑定
4. 高级设置（折叠）

## 6. 风险

- 如果继续把“系统唯一主 Agent”和“每个 Agent 归属项目”同时做成强规则，心智会冲突。
- 如果内置模板仍然和已创建 Agent 混在一个列表里，用户会继续把模板误当成已拥有对象。
- 如果编辑页仍保留大量自由文本输入，支持成本和误配概率都会持续偏高。

## 7. 结论

从产品和 UX 视角看，050 应解决的不是“模板编辑器不够美观”，而是对象模型表达错误：

- `Project` 应成为 Agent 管理的第一层边界
- 主 Agent、已创建 Agent、内置模板必须严格分层
- `Agents` 首页应回归列表管理
- 编辑体验应从技术字段表单改成结构化选择器

这是一项适合作为独立 Feature 推进的产品重整。
