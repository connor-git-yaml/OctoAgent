# Clarification Report: Feature 065 — 全面改进 Behavior 默认模板内容

**日期**: 2026-03-19
**输入制品**: `spec.md` (Draft)

## 1. 歧义检测

### [WARNING] AMB-01: FR-033 "合理水平" 的 40%-95% 区间缺乏下限强制性

**位置**: FR-033 — "目标区间：每个文件预算利用率 40%-95%"

**问题**: FR-033 同时使用了 MUST 级别和"目标区间"措辞，存在语义矛盾。MUST 意味着硬约束，但"目标区间"暗示是建议值。如果某个文件（如 HEARTBEAT.md，预算 1600 字符）内容域覆盖完整但只需 600 字符（37.5%），是否算违反了 FR-033？

**影响**: 实现阶段可能出现"为了凑字数而注水"的倾向，反而降低模板质量。

**AUTO-CLARIFIED**: 将 FR-033 理解为"SHOULD 达到 40% 以上，MUST 不超过 95%"。实际验收以各文件对应 FR 的内容域覆盖率为硬约束，字符利用率作为辅助指标。理由：内容域完整性比字符数更有意义；当前最高利用率仅 14.5%，40% 作为 SHOULD 目标已是 3-4 倍提升。

---

### [INFO] AMB-02: FR-022 "结论优先""不装懂""边界明确"是示例还是硬要求

**位置**: FR-022 — "覆盖'结论优先''不装懂''边界明确'等原则"

**问题**: "等原则"暗示这三条是示例而非穷举，但 Acceptance Scenario 4.1 要求"核心价值观（不少于 3 条）"。如果实现时用了不同的价值观措辞（如"坦诚"替代"不装懂"），是否满足 FR-022？

**AUTO-CLARIFIED**: 这三条是推荐的核心方向而非逐字硬要求。实现时只需覆盖语义等价的原则即可（如"坦诚面对不确定性"等价于"不装懂"）。验收标准以语义覆盖为准，不要求逐字匹配。

---

### [INFO] AMB-03: FR-018 "预期回答类型" 的含义不明确

**位置**: FR-018 — "每步明确：提问内容、预期回答类型、数据存储去向"

**问题**: "预期回答类型"可以理解为：(a) 数据类型（如字符串、枚举值），(b) 回答形式（如单选、自由文本、确认），(c) 举例说明预期答案。

**AUTO-CLARIFIED**: 理解为回答形式的简短说明，如"自由文本""单选""是/否"等。不需要程序化的类型声明，这是引导脚本而非 API contract。

## 2. 缺失检测

### [WARNING] MISS-01: 缺少对 BOOTSTRAP.md 引导步骤具体内容的约束

**位置**: FR-018 仅要求"不少于 4 步"，User Story 2 提到"称呼、偏好、Agent 性格等"

**问题**: 当前源码中 BOOTSTRAP.md 模板已包含 5 个引导项（称呼、默认 Agent 名称、性格偏好、时区/地点、长期偏好），但 FR-018 没有约束具体应包含哪些步骤。实现时可能遗漏关键步骤或增加不必要的步骤。

**AUTO-CLARIFIED**: 引导步骤至少应覆盖现有模板中已列出的 5 个维度（称呼、Agent 名称、性格偏好、时区/地点、长期偏好），可在此基础上增删但不得少于 4 步。新增步骤需要有对应的数据存储去向说明。理由：保持向前兼容，不丢失现有功能。

---

### [WARNING] MISS-02: 缺少 SC-005 的可操作测试方法

**位置**: SC-005 — "以默认模板创建的新 Agent，在首次交互中能正确识别自身角色...无需额外人工提示"

**问题**: SC-005 是一个定性的 LLM 行为验证标准，无法通过单元测试或字符串匹配自动验证。spec 中没有说明如何测试这条标准。这与其他 SC（SC-001 到 SC-004 均可自动化验证）形成反差。

**AUTO-CLARIFIED**: SC-005 作为集成验收的定性标准保留，但不纳入自动化测试范围。实际验收方式为：在开发完成后手动创建一个使用默认模板的新 Butler Agent，发送一条测试消息（如"你是谁？你能做什么？"），观察 Agent 回复是否体现角色认知和能力范围。此标准为 SHOULD 级别而非阻塞级别。理由：LLM 输出具有随机性，硬判定不可行。

---

### [INFO] MISS-03: 未提及模板内容中 Markdown 格式的使用规范

**位置**: 全局

**问题**: 当前 9 个模板均为纯文本字符串拼接，无 Markdown 标记（除 BOOTSTRAP.md 中有 `## 完成引导` 标题）。改进后的模板如果大幅使用 Markdown 结构（标题、列表、加粗等），Markdown 标记字符会占用预算但不直接携带语义信息。spec 未说明模板内容是否应使用 Markdown 格式。

**AUTO-CLARIFIED**: 模板内容可以使用轻量 Markdown 格式（如标题分区、无序列表）来提高可读性和结构化程度，但应避免过度装饰（如多级嵌套、表格等），因为 Markdown 标记会占用字符预算。参考现有 BOOTSTRAP.md 的 `## 完成引导` 风格。理由：行为文件最终注入 system prompt，结构化标记有助于 LLM 解析指令层级。

---

### [INFO] MISS-04: 未说明模板间的交叉引用策略

**位置**: 全局

**问题**: 9 个行为文件可能存在内容重叠。例如 AGENTS.md 提到"安全红线"，TOOLS.md 提到"secrets 安全边界"，SOUL.md 提到"边界明确"。改进后这些内容是否应互相引用（如 AGENTS.md 写"安全红线详见 TOOLS.md"），还是各自独立完整？

**AUTO-CLARIFIED**: 各文件保持独立完整，不使用交叉引用。理由：(a) 行为文件按 BehaviorLoadProfile 差异化加载，不保证所有文件同时出现在上下文中；(b) 交叉引用增加维护成本且可能出现死链；(c) 每个文件有独立的关注点和字符预算，重复少量关键原则（如安全红线）是可接受的。

## 3. 冲突检测

### [INFO] CONF-01: FR-035 与 Edge Case "模板内容与 system prompt 其他部分的冲突" 之间的处理缺失

**位置**: FR-035 vs Edge Cases 第 6 条

**问题**: FR-035 要求模板"反映 OctoAgent 实际架构能力"，Edge Case 提到"如果模板中的指令与 provider 层的 system prompt 指令矛盾，应以哪个为准"。但 spec 没有为此提供解决方案或验收标准。

**AUTO-CLARIFIED**: 行为文件在 system prompt 中优先级最高（insert 到 system_prompt[0]），因此模板设计时应避免与 provider 层指令产生矛盾，而非依赖优先级覆盖。具体做法：模板中不重复 provider 层已有的通用指令（如模型能力声明），只补充 provider 层无法覆盖的行为规范。理由：spec Edge Case 已明确 Behavior 优先级最高，但"避免冲突"优于"依赖覆盖"。

---

### 无冲突项

- FR 之间无逻辑矛盾：每个 FR 针对不同的 file_id，字符预算各自独立。
- FR 与 User Story 之间一致：每个 User Story 的 Acceptance Scenario 可追溯到对应的 FR。
- P1/P2 划分与 User Story 的优先级说明一致。

## 4. 可测试性检查

### [WARNING] TEST-01: SC-005 不可自动化测试

**详见 MISS-02**。SC-005 是唯一无法自动化验证的 Success Criteria。其他 SC-001 到 SC-004 均可通过单元测试或静态检查验证。

---

### [INFO] TEST-02: FR 中 "至少 N 条/项/个" 的断言需要内容解析

**位置**: FR-001(e 个内容域), FR-004(3 个区域), FR-007(4 个区域), FR-022(3 条价值观), FR-029(4 项清单)

**问题**: 这些 FR 使用了数量下限约束，验证时需要对模板内容进行结构化解析（如计算 Markdown 标题数或列表项数）。这不是不可测试，但需要约定内容的结构标记方式，以便测试代码能自动计数。

**AUTO-CLARIFIED**: 测试策略采用简单的子字符串/正则匹配验证内容域存在性，而非精确的结构解析。例如验证 AGENTS.md 包含"委派""安全红线""内存""A2A"等关键词，而非计算段落数。理由：模板是给 LLM 阅读的指令文本，过度结构化反而限制了内容质量。
