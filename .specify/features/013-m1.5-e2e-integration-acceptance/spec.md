## 前置条件（Pre-conditions）

本 Feature 为串行集成验收阶段，要求以下上游 Feature 全部已交付并通过各自验收标准：

| Feature | 名称 | 必须满足的门禁条件 |
|---------|------|------------------|
| Feature 008 | Orchestrator Skeleton | 消息路由协议包含版本化合约；路由决策、Worker 派发、Worker 回传三类系统事件已实现 |
| Feature 009 | Worker Runtime | Worker 可执行任务并回传产物；超时分级与取消机制已验证 |
| Feature 010 | Checkpoint & Resume | 检查点读取接口已实现；重复恢复幂等性与快照损坏降级已测试通过 |
| Feature 011 | Watchdog + Task Journal | 无进展检测器已实现；无进展告警事件写入已验证；默认检测阈值已配置 |
| Feature 012 | Logfire + Health/Plugin Diagnostics | 任务追踪标识符贯通埋点已完成；可观测后端不可用时的降级行为已验证 |

**门禁约束**（来自 `docs/m1.5-feature-split.md` §3.3）：

- GATE-M15-CONTRACT：Feature 008 路由协议未版本化，则 Feature 013 不得进入集成验收
- GATE-M15-RECOVERY：Feature 010 幂等恢复测试未通过，则 Feature 013 不得进入联调
- GATE-M15-WATCHDOG：Feature 011/012 无默认阈值且链路不可复盘，则 Feature 013 不得签字

## User Scenarios & Testing

### User Story 1 — 消息路由全链路可靠执行（Priority: P1）

作为 OctoAgent 用户，我希望提交一条任务消息后，系统能够可靠地完成从接收、决策、执行到返回结果的完整处理过程，以便我提交的每一个任务都能得到明确的处理结果，而不会因为系统内部组件的对接问题导致任务静默失败或无故卡住。

**Why this priority**: 这是 M1.5 最核心的验收条目（来自 Blueprint §14 条目 1）。若消息路由闭环不通，断点恢复、无进展检测、链路追踪均无实际意义。此条验收通过，是其他三条验收的前提基础。

**Independent Test**: 向系统提交一条消息，验证系统完整处理该消息、记录了全部处理步骤，并以成功状态结束任务。

**Acceptance Scenarios**:

1. **Given** 系统以真实组件（无测试替代）启动，**When** 通过消息接口提交一条低风险任务，**Then** 任务最终状态变为成功，系统记录了路由决策和执行过程，所有记录均关联到同一任务标识符。

2. **Given** 上述任务完成后，**When** 查询任务详情，**Then** 返回的执行结果非空，且与实际执行产物一致。

3. **Given** 系统以不依赖外部服务的方式运行，**When** 提交任务，**Then** 整条处理链路在 5 秒内完成，测试过程稳定可重复。

---

### User Story 2 — 系统中断后任务自动恢复（Priority: P1）

作为 OctoAgent 用户，我希望系统在意外中断（如进程崩溃、服务重启）后，能够自动从上次的中断点继续执行未完成的任务，而不是从头重新开始，以便用户不需要手动重试任务，已经完成的工作不会因为系统重启而白费。

**Why this priority**: 这是 M1.5 验收条目 2（Blueprint §14），也是"Durability First"Constitution 约束的可证明验证。任务断点恢复能力是系统可靠性的核心保证。

**Independent Test**: 预先记录任务执行进度，模拟系统意外中断，重新启动后验证任务从正确的中断点继续执行，而非重新开始。

**Acceptance Scenarios**:

1. **Given** 一个任务已记录了执行进度（包含中断前完成的步骤），**When** 模拟系统意外中断后重新恢复，**Then** 任务从上次记录的进度点继续执行，已完成的步骤不会重复执行，最终以成功状态结束。

2. **Given** 同一个任务的恢复操作被触发两次，**When** 第二次触发时任务已处于恢复完成状态，**Then** 第二次恢复操作不会产生重复的副作用，行为保持幂等。

3. **Given** 任务的执行进度记录已损坏或版本不兼容，**When** 系统尝试恢复该任务，**Then** 系统安全降级，任务以失败状态结束，并保留可查的失败原因记录，不影响系统整体稳定性。

---

### User Story 3 — 长时间无进展任务自动告警（Priority: P1）

作为 OctoAgent 用户，我希望当某个任务长时间没有任何进展时，系统能够自动检测到这一异常状态并生成告警，以便运维人员可以及时发现并处置卡死的任务，避免系统资源长期被占用而无实际产出。

**Why this priority**: 这是 M1.5 验收条目 3（Blueprint §14），是运行治理能力的核心证据。无进展任务检测直接关系到长任务失控风险（Blueprint §15）。

**Independent Test**: 创建一个已进入运行状态但停止产生进展的任务，配置极短的检测等待期，触发一次检测扫描，验证系统生成了无进展告警记录。

**Acceptance Scenarios**:

1. **Given** 一个任务处于运行状态且在配置的等待时限内没有产生任何进展记录，**When** 无进展检测服务执行一次扫描，**Then** 系统生成该任务的无进展告警记录，记录中包含对应的任务标识符。

2. **Given** 无进展告警已触发并进入冷却期，**When** 在冷却期未结束时再次执行检测扫描，**Then** 不会重复生成告警，冷却机制正常生效，避免告警风暴。

3. **Given** 无进展检测的等待阈值可由配置覆盖，**When** 将阈值设置为最小值后执行场景，**Then** 系统在最短等待时间后即触发告警，无需等待真实的时钟流逝。

### User Story 4 — 全链路执行过程完整可追溯（Priority: P1）

作为 OctoAgent 用户，我希望当任务出现问题时，能够追溯从消息接收到最终执行的每一个处理步骤，查看各环节的耗时和状态，以便在排查问题时能快速定位是哪一层出了问题，而不是面对一个黑盒。

**Why this priority**: 这是 M1.5 验收条目 4（Blueprint §14），也是"Observability is a Feature"Constitution 约束的可证明验证。链路可追溯性是可运维系统的基础要求。

**Independent Test**: 完成一次完整的消息处理请求，验证系统在接收层、路由决策层、Worker 执行层均保留了可关联的追踪记录，且各层记录可通过统一的任务标识符串联。

**Acceptance Scenarios**:

1. **Given** 系统以不对外上报追踪数据的方式运行，**When** 完成一次完整的消息接收到 Worker 执行的全链路请求，**Then** 系统在消息接收层、路由决策层、Worker 执行层均保留了追踪记录，所有记录携带相同的任务标识符，可以通过该标识符关联查询整条链路。

2. **Given** 上述追踪记录已生成，**When** 验证各层记录的关联关系，**Then** Worker 执行记录可追溯到路由决策记录，各层记录构成连续完整的处理链，链路无断裂。

3. **Given** 可观测后端（追踪数据上报服务）不可达，**When** 执行正常的消息路由任务，**Then** 主业务流程不受任何影响，任务正常完成，可观测能力不可用不会导致业务中断。

---

### User Story 5 — M1 已交付能力不因集成而退化（Priority: P2）

作为 OctoAgent 开发者，我希望在接入真实的 M1.5 完整依赖链路后，Feature 002 至 007 所覆盖的 M1 核心能力（模型供应商对接、认证鉴权、工具代理、技能执行、策略引擎、M1 端到端验收）仍然全部正常工作，以便确认新增的集成层没有破坏已经稳定交付的基础能力，M1 的交付成果得到保护。

**Why this priority**: 这是 M1.5 验收的隐性前提——新增集成不得引入回归。优先级低于四条 M1.5 核心验收条目，但同等重要：任何已有能力的退化都会导致 M1.5 验收无效。

**Independent Test**: 执行全量回归测试，确认 Feature 002 至 007 范围内的所有测试用例通过率保持 100%，无新增失败。

**Acceptance Scenarios**:

1. **Given** Feature 013 所有新增集成测试已完成，**When** 执行全量回归，**Then** Feature 002 至 007 范围内的已有测试全部通过，无新增失败用例。

2. **Given** 全量回归完成，**When** 查看测试覆盖报告，**Then** M1 已交付模块的测试覆盖率不低于回归前的基线值。

---

### User Story 6 — 产出 M1.5 验收报告为 M2 提供准入基线（Priority: P2）

作为架构负责人，我希望 Feature 013 产出一份结构化的 M1.5 验收报告，明确映射四条 M1.5 验收标准到可重现的测试证据，并附带技术风险清单，以便 M2 启动时有明确的准入基线，验收结论可传递给后续决策者，技术债务得到显式记录。

**Why this priority**: 这是交付层面的产出要求，没有报告则验收结论无法传递给后续决策者，M2 准入将缺乏依据。

**Independent Test**: 检查验收报告文件存在，且包含四条 M1.5 验收标准与测试证据的映射关系，以及技术风险状态的更新说明。

**Acceptance Scenarios**:

1. **Given** 所有端到端场景测试和回归测试执行完成，**When** 生成 M1.5 验收报告，**Then** 报告包含：(a) 四条 M1.5 验收标准与具体测试场景的映射关系，(b) 测试执行摘要（通过与失败数、覆盖情况），(c) 技术风险清单及当前状态（已缓解或遗留）。

2. **Given** 验收报告已生成，**When** 阅读报告，**Then** 报告中每条风险均有明确的状态标注（已缓解或遗留说明），不存在状态未知的技术风险。

### Edge Cases

- **上游检查点持久化能力缺失**：若 Feature 010 的检查点读取接口未正确实现或未接入持久化存储，User Story 2 将无法执行。此情形需在 Feature 013 工作开始前前置核查；若未实现，需在 Feature 013 内补充最小可用实现（关联 FR-006）。

- **无进展检测跨测试状态污染**：若无进展检测的冷却状态由全局单例维护，不同测试之间的冷却状态可能导致告警漏检。每个无进展检测测试场景应使用独立的冷却状态环境，或在测试结束后清理冷却状态。

- **恢复锁跨测试泄漏**：若任务恢复的互斥锁由类级别变量维护，测试异常退出可能留下残留锁，导致后续测试进入死锁。每个恢复场景测试应使用唯一的任务标识符，并确保测试结束后锁状态被清理。

- **追踪数据捕获工具不可用**：若用于验证全链路追踪的测试工具在当前运行时版本中不可用，User Story 4 的追踪断言将失败。此时应降级为备选的追踪数据捕获方案，不阻塞其他场景的验收，并在验收报告中注明降级情况。

- **全链路等待时序不稳定**：在慢速执行环境（如 CI 服务器）中，等待链路完成的固定时间可能不足。全链路测试应采用轮询加超时的等待方式，避免因时序不稳定导致的误判失败。

- **无进展检测调度与手动触发竞争**：若系统启动时同时开启了无进展检测的自动调度，调度器的后台扫描可能与手动触发的测试场景产生竞争，影响断言结果。无进展检测测试场景应在不启动自动调度的环境中执行，确保测试结果确定可控。

- **M1.5 门禁条件未满足**：若上游 Feature 008 至 012 存在接口未实现或测试未通过，Feature 013 的集成测试将因依赖缺失而大面积失败。此情形应记录在验收报告的风险清单中，并明确标注为"上游阻塞"，而非 Feature 013 自身问题。

## Requirements

### Functional Requirements

- **FR-001**: 系统 MUST 在 Feature 013 集成测试中接入真实组件依赖链路，去除 Feature 008 至 012 并行开发阶段引入的所有测试替代（mock），确保四条端到端验收场景均在真实运行状态下执行。[关联 US-1, US-2, US-3, US-4]

- **FR-002**: 系统 MUST 新增端到端验收场景 A（消息路由全链路），验证系统完整处理消息从接收到结果返回的全过程，记录路由决策和执行过程，任务以成功状态结束，所有记录关联同一任务标识符。[关联 US-1]

- **FR-003**: 系统 MUST 新增端到端验收场景 B（系统中断后恢复），通过模拟系统意外中断场景，验证系统能够从记录的中断点继续执行，已完成步骤不重复执行，恢复行为幂等。[关联 US-2]

- **FR-004**: 系统 MUST 新增端到端验收场景 C（无进展任务告警），通过配置最短检测阈值并触发无进展扫描，验证系统对超时无进展的任务生成告警记录，冷却机制防止重复告警。[关联 US-3]

- **FR-005**: 系统 MUST 新增端到端验收场景 D（全链路追踪贯通），验证系统在消息接收、路由决策、Worker 执行三个处理层均保留了关联的追踪记录，各层记录通过统一任务标识符可完整串联。[关联 US-4]

- **FR-006**: 系统 MUST 在 Feature 013 工作开始前核查检查点持久化读取能力的实现完整性；若未实现，需在 Feature 013 内补充最小可用实现，且该实现须通过场景 B 的验收断言，方可视为场景 B 完成。[关联 US-2]

- **FR-007**: 系统 MUST 执行 Feature 002 至 007 范围内的全量回归，确认新增集成层没有破坏 M1 已交付能力，测试通过率相较 M1 基线无新增失败。[关联 US-5]

- **FR-008**: 系统 MUST 产出 M1.5 验收报告，包含四条 M1.5 验收标准与测试场景的映射关系、测试执行摘要，以及技术风险清单的当前状态。[关联 US-6]

- **FR-009**: 系统 SHOULD 保证端到端验收测试在不同执行环境（本地开发机、CI 服务器）下均产出稳定一致的结果，不因执行环境差异导致误判失败。[关联 US-1, US-2, US-3, US-4]

- **FR-010**: 系统 SHOULD 保证无进展检测的各验收场景之间互不干扰，一个场景中产生的检测状态不影响其他场景的告警判断结果。[关联 US-3]

- **FR-011**: 系统 SHOULD 保证系统中断恢复的各验收场景之间互相隔离，一个场景的恢复操作不对其他场景的任务状态产生残留影响。[关联 US-2]

- **FR-012**: 系统 MAY 在全链路追踪验证工具不可用时降级为备选追踪数据捕获方案完成验证，降级情况须在验收报告中注明。[关联 US-4]

- **FR-013**: 系统 MUST NOT 在 Feature 013 中引入任何新的业务能力或架构组件；本 Feature 仅负责集成验收，不扩展功能范围。[来自 `docs/m1.5-feature-split.md` §3.2]

### Key Entities

- **M1.5 端到端验收测试套件**：覆盖四个端到端验收场景（消息路由闭环、系统中断恢复、无进展告警、全链路追踪）的集成测试集合，各场景独立可运行，使用真实组件（无测试替代）。

- **M1.5 验收报告**：结构化文档，包含四条 M1.5 验收标准与测试证据的映射关系、测试执行摘要（通过与失败数、覆盖情况），以及技术风险清单的当前状态说明。

- **集成测试基础设施**：为四条端到端场景提供测试基础环境，包含系统中断场景预置能力、无进展检测场景的隔离运行能力，以及全链路追踪数据的捕获能力。

## Success Criteria

### Measurable Outcomes

- **SC-001（消息路由全链路可靠执行）**: 场景 A 验收通过，系统完整处理消息，记录路由决策和执行全过程，所有记录关联同一任务，任务以成功状态结束，处理链路在 5 秒内完成。[关联 FR-002, US-1]

- **SC-002（系统中断后任务自动恢复）**: 场景 B 验收通过，系统在模拟意外中断后成功从中断点继续执行，已完成步骤不重复执行，连续两次触发恢复的结果一致（幂等），不产生重复副作用。[关联 FR-003, FR-006, US-2]

- **SC-003（长时间无进展任务自动告警）**: 场景 C 验收通过，系统对超出配置等待时限的无进展任务生成告警记录，冷却机制在重复触发时生效，不产生重复告警。[关联 FR-004, US-3]

- **SC-004（全链路执行过程完整可追溯）**: 场景 D 验收通过，系统在消息接收、路由决策、Worker 执行三个处理层均保留了追踪记录，所有记录通过统一任务标识符可完整串联，链路无断裂，可观测后端不可用不影响主业务流程。[关联 FR-005, US-4]

- **SC-005（M1 已交付能力不退化）**: 全量回归执行后，Feature 002 至 007 范围内的已有测试全部通过，无新增失败，M1 已交付模块的测试覆盖不低于回归前基线。[关联 FR-007, US-5]
