## 1. 背景与目标

### 1.1 M1 收口基线

基于当前 `master`（`111b8da`）：

- M1 核心链路（Provider/Auth/Tooling/Skill/Policy/007 集成）已形成闭环。
- 007 真实联调已验证：`SkillRunner -> ToolBroker -> Policy -> Approval`。
- 主聊天链路尚未切换为 Orchestrator/Worker 分层运行平面（Blueprint 明确放在 M1.5 评估）。

### 1.2 M1.5 目标（Blueprint §14）

M1.5 定位为“最小 Agent 闭环（2 周）”：

- 简化版 Orchestrator（目标理解 + 单活 Worker 直连 + 可扩展派发契约）
- 通用 Worker 框架（Free Loop）
- Checkpoint 恢复（从最后成功点继续）
- Watchdog 无进展检测
- Logfire 可观测链路

## 3. 并行拆分方案（M1.5 = 6 个 Feature）

### 3.1 依赖图

```text
M1 基线（已完成）
   │
   ├── Track A: 控制平面
   │   ├── Feature 008: Orchestrator Skeleton
   │   └── Feature 009: Worker Runtime + Docker/Timeout/Profile
   │
   ├── Track B: 可恢复与治理
   │   ├── Feature 010: Checkpoint & Resume
   │   └── Feature 011: Watchdog + Task Journal + Drift Detector
   │
   └── Track C: 可观测与诊断
       └── Feature 012: Logfire + Health/Plugin Diagnostics

全部汇合
   └── Feature 013: M1.5 E2E 集成验收
```

### 3.2 并行化原则

1. 先冻结接口契约，再并行编码：008 先输出 `Orchestrator<->Worker` 协议，009/010/011/012 并行。  
2. Track B 与 Track C 不阻塞 Track A：Checkpoint/Watchdog/Observability 可先以事件契约对接。  
3. 013 仅做真实联调与验收，不引入新能力，避免范围膨胀。

### 3.3 调研复核后的三条必改约束（Must）

1. **控制平面契约必须可扩展**：M1.5 虽然是单 Worker，但 `DispatchEnvelope` 不能硬编码；至少包含 `contract_version`、`route_reason`、`worker_capability`、`hop_count/max_hops`。  
2. **恢复语义必须幂等且可证明一致**：checkpoint 写入与关键事件写入必须定义事务边界与冲突处理，避免恢复后重复执行副作用。  
3. **Watchdog 必须给出默认阈值并可观测**：必须定义 heartbeat 周期、无进展窗口、cooldown，且保证 `task_id/trace_id/span_id` 全链路透传可验证。

## 4. Feature 详细拆解

### Feature 008：Orchestrator Skeleton（单 Worker）

**一句话目标**：落地最小 Orchestrator 控制平面，接收任务并派发到单 Worker。

**覆盖需求**：
- FR-A2A-1（目标理解、派发、监督、门禁）
- §14 M1.5 验收条目 1（用户消息到 Worker 结果回传）

**参考实现证据（project-context 对齐）**：
- `/_references/opensource/agent-zero/python/tools/call_subordinate.py`（委派与回收控制）
- `/_references/opensource/agent-zero/python/helpers/subagents.py`（子代理注册与覆盖合并）
- `/_references/opensource/agentstudio/backend/src/services/taskExecutor/index.ts`（执行器工厂与单例初始化）
- `/_references/opensource/pydantic-ai/docs/multi-agent-applications.md`（delegation/handoff 模式）

**任务拆解**：
- F008-T01：定义 `OrchestratorRequest/DispatchEnvelope/WorkerResult` 数据模型与协议（包含 `contract_version`、`route_reason`、`worker_capability`、`hop_count/max_hops`）。
- F008-T02：实现单 Worker 路由器（先 rule-based，预留策略入口）。
- F008-T03：实现 `Task -> Orchestrator -> Worker` 最小执行循环。
- F008-T04：新增事件类型：`ORCH_DECISION`、`WORKER_DISPATCHED`、`WORKER_RETURNED`。
- F008-T05：接入 Policy Gate（仅高风险动作 gate，不做全量策略重写）。
- F008-T06：单元测试（路由、派发、失败回传）。
- F008-T07：集成测试（用户消息到 Worker 回传）。

**验收标准**：
- 用户消息可由 Orchestrator 正确派发到目标 Worker；
- Worker 结果可回传并落盘事件链；
- 失败可归类为可重试/不可重试；
- `DispatchEnvelope` 协议具备版本化与跳数保护（`hop_count <= max_hops`）。

### Feature 009：Worker Runtime + Docker + Timeout/Profile

**一句话目标**：构建 Worker Free Loop 运行时，激活 Docker 执行与 `privileged` profile。

**覆盖需求**：
- FR-A2A-2（自治执行、可中断、回传）
- §8.5.4（`privileged` 在 M1.5 激活）
- §8.5.6（细粒度超时分级，优先用于执行工具）

**参考实现证据**：
- `/_references/opensource/agentstudio/backend/src/services/taskExecutor/taskWorker.ts`（worker 隔离执行与结果回传）
- `/_references/opensource/agent-zero/python/helpers/docker.py`（容器生命周期与可用性探测）
- `/_references/opensource/openclaw/src/agents/pi-tools.before-tool-call.ts`（hook 化执行前拦截思路）

**任务拆解**：
- F009-T01：定义 WorkerSession（loop_step、budget、tool_profile、state）。
- F009-T02：实现 Free Loop 驱动器（max_steps、budget、中断信号）。
- F009-T03：实现 Docker 执行后端接入（默认隔离，策略可配）。
- F009-T04：激活 `privileged` profile，并绑定显式授权策略。
- F009-T05：引入分层超时（first_output/between_output/max_exec）。
- F009-T06：中断/取消语义（RUNNING -> CANCELLED/FAILED）与事件。
- F009-T07：集成测试（长任务、超时、取消、权限拒绝）。

**验收标准**：
- Worker 能在隔离环境中执行工具链并回传产物；
- `privileged` 工具仅在策略授权后可用；
- 超时分级与取消机制生效。

### Feature 010：Checkpoint & Resume Engine

**一句话目标**：实现节点级 checkpoint 与恢复，不再全量重跑。

**覆盖需求**：
- FR-TASK-4（Checkpoint + 从最后成功点恢复）
- §14 M1.5 验收条目 2（中断恢复）

**参考实现证据**：
- `/_references/opensource/pydantic-ai/docs/durable_execution/overview.md`（耐久执行总体模式）
- `/_references/opensource/pydantic-ai/pydantic_graph/pydantic_graph/persistence/file.py`（snapshot 持久化与并发锁）
- `/_references/opensource/agent-zero/python/helpers/task_scheduler.py`（任务状态与计划持久化）

**任务拆解**：
- F010-T01：定义 `CheckpointSnapshot` 模型（task_id/node_id/state/ts/version）。
- F010-T02：持久化层（SQLite 表 + 索引 + 幂等写 + schema version）。
- F010-T03：在 Worker/Skill Pipeline 节点切换处写 checkpoint。
- F010-T04：恢复器（加载最后成功 checkpoint 并重建执行上下文）。
- F010-T05：恢复冲突策略（版本不一致、快照损坏、缺失回退）。
- F010-T06：定义 checkpoint 与关键事件写入事务边界（原子提交或可恢复补偿）。
- F010-T07：故障注入测试（进程重启、快照损坏、重复恢复、恢复幂等）。

**验收标准**：
- 中断后可从最近成功节点继续；
- 快照损坏可降级为安全失败并可重试；
- 恢复过程全链路可观测；
- 重复恢复不会重复执行已落盘副作用（幂等性校验通过）。

### Feature 011：Watchdog + Task Journal + Drift Detector

**一句话目标**：实现任务运行治理，检测无进展/漂移并触发告警或自动动作。

**覆盖需求**：
- §14 M1.5 验收条目 3（watchdog）
- Blueprint 风险治理 §15（长任务失控）
- 延续 007 的运行治理增强方向（Task Journal / drift）

**参考实现证据**：
- `/_references/opensource/agent-zero/python/helpers/task_scheduler.py`（计划任务状态管理与运行态更新）
- `/_references/opensource/openclaw/src/agents/pi-tools.before-tool-call.ts`（循环检测分级）
- `/_references/opensource/openclaw/docs/gateway/doctor.md`（健康检查与修复建议输出）

**任务拆解**：
- F011-T01：新增事件：`TASK_HEARTBEAT`、`TASK_MILESTONE`、`TASK_DRIFT_DETECTED`。
- F011-T02：Task Journal 投影视图（运行中/疑似卡死/漂移/待审批）。
- F011-T03：无进展检测器（基于事件时间窗与进度增量）。
- F011-T04：漂移检测器（状态机漂移、长时间 pending、重复失败模式）。
- F011-T05：策略动作（提醒/降级/暂停/取消）与可配置阈值（提供默认值与配置项）。
- F011-T06：E2E 测试（模拟卡死、循环、状态漂移）。

**验收标准**：
- Watchdog 能在阈值内发现无进展任务；
- 诊断输出包含可执行修复建议；
- 告警动作可审计可回放；
- 默认阈值生效（示例：heartbeat=15s，no_progress=3 个周期，cooldown=60s）。

## 6. 建设性意见（对现有方案的调整建议）

1. **建议把“单 Worker 直连”实现为“单活 Worker + 可扩展派发接口”**  
理由：Blueprint M1.5 写“简化版单 Worker”，但 FR-A2A-1 要求“Worker 选择与派发”。若直接硬编码单 Worker，M2 多 Worker 会发生重构返工。

2. **建议把插件加载隔离设为 M1.5 的“should-have”而非“must-have”**  
理由：它来自 §8.5.6 演进项，不是 §14 M1.5 四条硬验收。可放入 Feature 012，若时间吃紧可降级为诊断先行、隔离后补。

3. **建议 checkpoint 先以“Worker 级快照”落地，再细化到 Graph 节点**  
理由：FR-TASK-4 要求 node_id + state snapshot，但当前主链路刚完成 M1 收口；先做 Worker 级 checkpoint 能更快达成可恢复闭环，再增量细化。

## 7. 需你确认的门禁（Design Gate）

- **GATE-M15-SCOPE（建议默认 A）**
  - A（推荐）：M1.5 先交付 008-013 主线，插件隔离做“诊断先行、进程隔离可后补”。
  - B（激进）：M1.5 同时完成插件进程隔离全量落地（增加实现风险与工期波动）。

- **GATE-M15-CONTRACT（建议默认通过）**
  - 008 若未包含 `contract_version` + `hop_count/max_hops`，009/013 不得进入集成验收。

- **GATE-M15-RECOVERY（建议默认通过）**
  - 010 若未通过“重复恢复幂等 + 快照损坏降级”测试，011/013 不得进入联调。

- **GATE-M15-WATCHDOG（建议默认通过）**
  - 011/012 若未给出默认阈值且无法在 Logfire/OTel 复盘判定链路，013 不得签字。

如果你不特别指定，默认采用 A。
