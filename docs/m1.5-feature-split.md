## 3. 并行策略复盘

### 3.1 实际依赖图

```text
M1 基线（已完成）
   │
   ├── Track A: 控制平面
   │   ├── Feature 008 ✅ Orchestrator Skeleton
   │   └── Feature 009 ✅ Worker Runtime + Docker/Timeout/Profile
   │
   ├── Track B: 可恢复与治理
   │   ├── Feature 010 ✅ Checkpoint & Resume
   │   └── Feature 011 ✅ Watchdog + Task Journal + Drift Detector
   │
   ├── Track C: 可观测与诊断
   │   └── Feature 012 ✅ Logfire + Health/Plugin Diagnostics
   │
   ├── Track D: DX 收口
   │   └── Feature 014 ✅ Unified Model Config
   │
   └── 汇合
       └── Feature 013 ✅ M1.5 E2E 集成验收
```

### 3.2 这次并行拆分哪些做对了

1. **先冻结 008 协议，再并行 009/010/012** 这个顺序是正确的。控制平面 contract 先定住后，其他 Track 才没有反复回改。
2. **011 放在 010 稳定后补核心检测逻辑** 是正确的。Watchdog 的无进展/漂移检测确实依赖 checkpoint 与事件基线稳定。
3. **013 只做集成，不引入新能力** 避免了验收阶段 scope 膨胀，这一点后续 M2 必须继续保持。
4. **014 作为 M1.5 收口 Feature 单独交付** 是必要的。否则 M2 的 onboarding 仍会建立在分散配置的旧现实上。

### 3.3 这次并行拆分暴露出的不足

1. `docs/m1.5-feature-split.md` 没有随着交付推进同步回写，导致文档长期停留在“待启动 008”。
2. 原拆分偏重内核能力，对“新用户怎么装好、怎么从 Telegram 操作、怎么做自助恢复”覆盖不足。
3. 014 这类“产品可用性收口”没有在初版里程碑拆分中显式占位，容易被误判成杂项而推迟。

## 5. M1.5 之后真正留下的缺口

这些不是 M1.5 失败，而是完成内核闭环后暴露出的**下一阶段真实短板**：

1. **首次使用路径仍不够完整**
   - 014 解决了模型配置碎片化，但还没有把 `octo config`、`octo doctor --live`、渠道接入、首条消息验证串成一次可恢复的 onboarding。
2. **Web 与 Telegram 的操作者体验还没有真正打平**
   - 审批、告警、重试、取消、pending 队列、过期时间等操作，需要统一成一个 operator inbox，而不只是散落在不同页面/事件里。
3. **自助恢复与迁移能力缺少产品化入口**
   - Blueprint 已有底层 backup/restore 策略，但用户侧没有 backup/export/restore dry-run 的稳定入口。

以上缺口全部转入 M2，并在 `docs/m2-feature-split.md` 中拆成独立 Feature。
